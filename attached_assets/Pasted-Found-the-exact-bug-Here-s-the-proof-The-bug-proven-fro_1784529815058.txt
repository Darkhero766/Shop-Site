## The bug — proven from code

In `buyer-auth-context.tsx` the `init()` function runs on mount:

```typescript
async function init() {
  const { data: { session } } = await buyerSupabase.auth.getSession()
  setBuyerSession(session)           // ✅ session ID restored
  if (session?.user?.id) {
    await fetchProfile(session.user.id) // fetches from buyers table
  }
}
```

Then `fetchProfile` does this:
```typescript
const { data, error } = await buyerSupabase
  .from("buyers")
  .select("*")
  .eq("id", userId)
  .maybeSingle()
```

**The problem is the RLS policy on your `buyers` table.** When the page reopens, the Supabase client restores the session token from localStorage — but the JWT token needs a moment to be recognized by Supabase's REST API. The `fetchProfile` query fires before the token is fully active on the server side, gets blocked by RLS (`auth.uid() = id` fails because uid isn't loaded yet), returns `null`, and sets `buyerProfile = null`.

**Result: session = logged in ✅, profile = null ❌**

## Also run this in Supabase SQL Editor right now

```sql
drop policy if exists "Buyers can read own profile" on buyers;
drop policy if exists "Buyers can view own profile" on buyers;

create policy "Buyers can read own profile"
on buyers for select
using (auth.uid() = id);

drop policy if exists "Buyers can insert own profile" on buyers;
create policy "Buyers can insert own profile"
on buyers for insert
with check (true);

drop policy if exists "Buyers can update own profile" on buyers;
create policy "Buyers can update own profile"
on buyers for update
using (auth.uid() = id);
```
